**دراسة فنية لتطوير نظام أرشفة إلكتروني لقسم ضمان الجودة وتقييم الأداء**

تقدم هذه الدراسة البنية التحتية، المعمارية البرمجية، وآلية عمل نظام الأرشفة الإلكتروني المطلوب، مع مراعاة متطلبات استخراج البيانات بالذكاء الاصطناعي، التكامل مع Google Workspace، ونظام التتبع الدقيق للوثائق الرسمية.

### 1. المعمارية التقنية المقترحة (Tech Stack)

لضمان الأداء العالي، الأمان، وسهولة الصيانة، يوصى بالتقنيات التالية:

* **الواجهة الأمامية (Frontend):** إطار عمل Vue.js (أو Nuxt.js) لبناء واجهة مستخدم تفاعلية وسريعة (SPA).
* **الواجهة الخلفية (Backend):** لغة Go (Golang) نظراً لكفاءتها العالية في معالجة الملفات، تنفيذ المهام المتزامنة (مثل ضغط الملفات وعمليات OCR)، وسرعة الاستجابة.
* **قاعدة البيانات (Database):** PostgreSQL. تعتبر الخيار الأمثل لدعمها المتقدم لبيانات JSONB (الضرورية للحقول المخصصة المتغيرة) وقدرتها على إدارة العلاقات المعقدة بكفاءة.
* **التخزين (Storage):** Google Drive API باستخدام حساب خدمة (Service Account) لمركزية التخزين، مع تخزين مسارات الملفات (File IDs) في قاعدة البيانات.

### 2. تفصيل الميزات والوظائف الأساسية

**أ. التكامل مع Google Drive وإدارة الملفات:**

* **الرفع والتصفح:** يتم رفع الملفات (PDF, Word, Images) عبر الواجهة الخلفية إلى Google Drive. الواجهة الأمامية تعرض هيكل شجري أو شبكي يعتمد على البيانات الوصفية لتصفح الملفات.
* **الضغط المسبق (Pre-compression):** * **الصور:** استخدام مكتبات مثل `libjpeg-turbo` أو `TinyGo` لتقليل الحجم مع الحفاظ على دقة القراءة.
* **ملفات PDF:** استخدام أدوات مثل `Ghostscript` برمجياً لضغط ملفات PDF قبل إرسالها إلى Drive.

متوافق مع الهاتف بحيث يمكن استخدام الهاتف لالتقاط الصور ورفعها بشكل مباشر 
* **المشاركة:** إنشاء روابط Google Drive بصلاحية (Read-only) وإتاحتها للمستخدمين المصرح لهم أو كروابط عامة مؤمنة بكلمة مرور.

**ب. البيانات الوصفية (Metadata) والحقول المخصصة:**

* بناء نظام Dynamic Schema باستخدام حقول JSONB في قاعدة البيانات، مما يتيح للمسؤول إضافة حقول جديدة (مثل: رقم الإضبارة، نوع الكابينة، تصنيف سري/عادي) دون الحاجة لتعديل هيكل قاعدة البيانات.

**ج. المعالجة الذكية (OCR) وتحليل البيانات (AI):**

* **الاستخراج النصي (OCR):** دمج محرك Tesseract OCR (مفتوح المصدر ومجاني) أو Google Cloud Vision لاستخراج النص الخام من الصور وملفات PDF الممسوحة ضوئياً.
* **تحليل البيانات:** يتم إرسال النص المستخرج إلى نموذج لغوي (LLM).
* **الخيارات:** يمكن استخدام واجهة برمجة تطبيقات Gemini، أو DeepSeek، أو تشغيل نموذج محلي مجاني تماماً مثل Llama 3 عبر Ollama لضمان خصوصية البيانات وعدم خروجها من خوادم الجامعة.
* **الآلية:** يتم توجيه النموذج (Prompt Engineering) لاستخراج كائن JSON دقيق يحتوي على: `{"عنوان_الكتاب": "...", "الجهة_المصدرة": "...", "رقم_العدد": "...", "تاريخ_الكتاب": "..."}` وملء الحقول تلقائياً ليراجعها المستخدم قبل الحفظ.



**د. المصادقة وصلاحيات الوصول (Authentication & Authorization):**

* **Google Workspace SSO:** الاعتماد على بروتوكول OAuth 2.0 لتسجيل الدخول باستخدام نطاق الجامعة (مثال: `@university.edu.iq`).
* **الدخول المحلي:** توفير نظام (Username/Password) للمستخدمين من خارج النطاق إذا لزم الأمر، مع واجهة للآدمن لتفعيل أو تعطيل هذا الخيار.

**هـ. سجل التدقيق (Audit Log) والحذف المنطقي (Soft Delete):**

* **Soft Delete:** لا يتم تنفيذ أمر `DELETE` فعلي في قاعدة البيانات. يتم استخدام حقل `deleted_at`. الملفات أو البيانات "المحذوفة" تختفي من واجهات المستخدمين العاديين وتنتقل إلى "سلة المهملات" الخاصة بالآدمن.
* **سجل التدقيق:** جدول قاعدة بيانات غير قابل للتعديل (Append-only) يسجل كل حركة (إضافة، تعديل، حذف، تنزيل) معرّفة بـ (User ID, Action, Timestamp, IP Address).

**و. تتبع العلاقات والكتب الرسمية:**

* **الأشخاص ذوو العلاقة:** إنشاء جدول `Persons` وجدول وسيط `Document_Persons` (علاقة متعدد إلى متعدد). يتيح ذلك ربط وثيقة واحدة بعدة أشخاص، وعند البحث عن اسم أكاديمي أو موظف، يظهر سجل كامل بجميع الوثائق المرتبطة به.
* **تتبع الوارد والصادر:** إضافة حقول لتحديد نوع الكتاب (صادر من الجامعة، وارد إلى الجامعة من وزارة/مؤسسة أخرى).
* **الوارد المتعدد (Multi-level Routing):** تصميم نظام تتبع للمسار (Workflow) يوضح نقطة دخول الكتاب (مثلاً: وارد مكتب رئيس الجامعة -> محال إلى قسم ضمان الجودة).

**ز. التصدير والتقارير:**

* توفير زر للتصدير يقوم بتوليد ملف Excel (باستخدام مكتبة مثل `excelize` في Go) يحتوي على البيانات الوصفية للوثائق المحددة.
* تجميع الملفات الفعلية المرتبطة بهذه الوثائق من Google Drive وضغطها في ملف ZIP واحد لتنزيله مع ملف Excel.

### 3. التحسينات المقترحة لبيئة الإنتاج (Production Readiness)

لتحويل هذا النظام إلى منتج احترافي يعتمد عليه على مستوى الجامعة، يجب تضمين التحسينات التالية في خطة التطوير:

1. **نظام الطوابير للمهام الثقيلة (Message Queuing):**
عمليات الـ OCR، تحليل الذكاء الاصطناعي، وضغط الملفات هي عمليات تستهلك وقتاً وموارد. يجب عدم تنفيذها بشكل متزامن مع طلب المستخدم لكي لا يتجمد النظام (Timeout). يُنصح باستخدام نظام طوابير (مثل RabbitMQ أو Asynq في Go) لمعالجة هذه المهام في الخلفية (Background Workers) وإشعار المستخدم عند اكتمال التحليل وتوفر البيانات.
2. **محرك بحث متقدم (Full-Text Search):**
البحث التقليدي في قاعدة البيانات قد يكون بطيئاً ومحدوداً. يوصى بدمج محرك بحث مثل Elasticsearch أو Meilisearch. هذا سيسمح بالبحث ليس فقط في البيانات الوصفية، بل داخل النص الكامل للوثائق (المستخرج عبر OCR) بسرعة فائقة، مع دعم التصحيح الإملائي والبحث المتقارب.
3. **إدارة الصلاحيات المبنية على الأدوار (RBAC):**
تطبيق نظام صلاحيات دقيق يحدد من يمكنه (القراءة، الإضافة، التعديل، الحذف المنطقي، إدارة النظام، تصفح سجل التدقيق). وتقسيم الصلاحيات بناءً على الأقسام (مستخدم في الموارد البشرية لا يرى ملفات ضمان الجودة السريّة).
4. **التخزين المؤقت (Caching):**
استخدام Redis لتخزين البيانات الوصفية المتكررة الاستعلام (مثل قوائم التصنيفات، هيكلية الإضبارات، وبيانات المستخدمين) لتقليل العبء على قاعدة البيانات وتسريع استجابة الواجهة.
5. **أمان النظام (Security Hardening):**
* تشفير البيانات الحساسة داخل قاعدة البيانات (Data at Rest).
* تطبيق Rate Limiting لمنع هجمات حجب الخدمة (DDoS) ومحاولات التخمين على صفحة تسجيل الدخول المحلية.
* إعداد نسخ احتياطي آلي (Automated Backups) لقاعدة البيانات بشكل يومي.


6. **النشر باستخدام الحاويات (Containerization):**
تغليف التطبيق (الواجهة، الخلفية، قاعدة البيانات، خدمات OCR) باستخدام Docker و Docker Compose، مما يسهل نشر النظام على خوادم الجامعة المركزية (مثل بيئة Proxmox) ويضمن تطابق بيئة التطوير مع بيئة الإنتاج.
